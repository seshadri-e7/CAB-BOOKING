                             USER FLOW

## Project Title

**UCAB – Cab Booking System (MERN Stack)**

## Objective

The User Flow document maps the detailed, step-by-step navigation paths and interaction logic for all actors within the UCAB application. By analyzing the workflow of Riders, Drivers, and Admins, this document ensures a seamless user experience (UX) and guides the development of frontend routing and backend API integration.

---

# Introduction

The UCAB application relies on synchronous state transitions between three core roles:
1. **Rider (User)**: Initiates bookings, tracks rides, and processes payments.
2. **Driver**: Moderates status (online/offline), accepts trip assignments, and drives routes.
3. **Admin**: Monitors operations, manages system data, and configures base system values.


---




# 1. Rider (User) Journey Flow

The Rider's journey spans from launching the application, selecting locations, choosing cab types, tracking the ride, to paying and completing the trip.

### Step-by-Step Flow

#### 1. Authentication & Onboarding
* User launches the React client.
* System checks token storage for active sessions.
  * *If session is expired/missing*: User is redirected to **Login/Signup** page.
  * *If session is active*: User lands on the **Rider Dashboard**.

#### 2. Booking a Ride
* User enters **Pickup Location** and **Drop-off Location** (supported by Google Maps Autocomplete).
* System calls the API to calculate coordinates and estimate distances.
* System displays estimated fares for different vehicle tiers: **Mini**, **Sedan**, **SUV**, **Premium**.
* User selects the desired cab category and clicks **Confirm Booking**.

#### 3. Matching & Dispatch
* Ride booking state is set to `PENDING`.
* System queries available drivers within a 5km radius.
* System sends WebSocket notifications to matched drivers.
  * *If driver accepts*: State transitions to `ACCEPTED`. Rider is shown driver profile, vehicle details, and real-time map location.
  * *If no driver accepts within 60s*: System prompts rider to retry or raise fare.

#### 4. Ride In-Progress
* Rider receives real-time coordinate pings as the driver approaches.
* When driver arrives, status updates to `ARRIVED` (Rider is notified).
* Driver inputs start OTP/code, updating ride status to `IN_TRANSIT`.
* Rider tracks the moving cab coordinates on the dashboard map.

#### 5. Destination & Payment
* Upon arrival at drop-off, driver completes ride. Status changes to `COMPLETED`.
* System renders the **Payment Screen** on the Rider app.
* Rider pays via online gateway or wallet.
* System displays transaction receipt.
* Rider completes a **Rate & Review** form for the driver.

---

# 2. Driver Journey Flow

The Driver's journey focuses on toggle status management, accept/reject decisions, navigation, status updates, and earnings tracking.

### Step-by-Step Flow

#### 1. Go Online / Offline
* Driver logs in and toggles state to **Online**.
* System begins registering driver's live coordinate updates inside the database.

#### 2. Handling Booking Requests
* Incoming booking request is pushed via WebSockets (showing pickup point, drop-off point, and fare estimate).
* Driver has 30 seconds to **Accept** or **Reject** the request.
  * *If Rejected*: Driver remains online for other rides.
  * *If Accepted*: System binds driver to `RideID` and reveals rider profile details.

#### 3. Passenger Pickup & Trip
* Driver navigates to the pickup coordinates using in-app map guidance.
* Upon arrival, Driver clicks **Arrived**.
* Once passenger boards, Driver verifies the trip start code and clicks **Start Ride**.

#### 4. Drop-off & Receipt
* Driver drives to the destination.
* Upon arriving, Driver clicks **End Ride**.
* Driver waits for payment confirmation screen and rates the rider.
* Driver accesses their **Earnings Screen** to view updated daily balances and trip statistics.

---

# 3. Admin Journey Flow

The Admin oversees operational state controls, monitors active rides, manages users, and handles financials.

### Step-by-Step Flow

#### 1. Portal Access
* Admin authenticates via the secure Admin Login panel.
* Lands on **Admin Dashboard** showing KPI widgets (Total Active Rides, Total Online Drivers, System Revenue, Active Disputes).

#### 2. Account Moderation
* Admin reviews pending Driver applications (checking uploaded License and Vehicle papers).
* Admin approves drivers, shifting their status to active.
* Admin manages Rider bans or reviews driver complaints.

#### 3. Pricing & Operational Control
* Admin adjusts pricing multipliers (e.g., setting surge rates during rush hour).
* Admin monitors active ride tracks globally on a map interface.

# Visual Workflows

### Rider Navigation Flowchart (Mermaid)

```mermaid
graph TD
    Start([Launch App]) --> AuthCheck{Session Active?}
    AuthCheck -- No --> Register[Register/Login]
    Register --> Dashboard
    AuthCheck -- Yes --> Dashboard[Rider Dashboard]
    
    Dashboard --> LocationSearch[Enter Pickup & Drop Locations]
    LocationSearch --> CabSelect[Select Cab Tier & View Fares]
    CabSelect --> ConfirmBooking[Click Confirm Booking]
    
    ConfirmBooking --> MatchSearch{Driver Found?}
    MatchSearch -- No --> Retry[Prompt Retry / Timeout]
    Retry --> LocationSearch
    MatchSearch -- Yes --> Accepted[Driver Details & ETA Shown]
    
    Accepted --> Pickup[Driver Arrives & Inputs Code]
    Pickup --> Transit[In Transit / Real-time Map Tracking]
    Transit --> Destination[Arrive at Destination]
    
    Destination --> Payment[Process Payment / Wallet Checkout]
    Payment --> Feedback[Rate & Review Driver]
    Feedback --> End([Trip Completed])
```

### Driver State Machine Flowchart (Mermaid)

```mermaid
graph TD
    Start([Driver Login]) --> StatusToggle[Toggle Status to Online]
    StatusToggle --> Standby[Standby / Listen for Requests]
    
    Standby --> RequestIn{Incoming Ride Request?}
    RequestIn -- No/Timeout --> Standby
    RequestIn -- Yes --> Decision{Accept Ride?}
    
    Decision -- No/Ignore --> Standby
    Decision -- Yes --> RouteToPickup[Navigate to Pickup Coordinate]
    
    RouteToPickup --> Arrived[Click Arrived at Pickup]
    Arrived --> OTPVerify[Verify Start Code]
    OTPVerify --> InTransit[Start Ride / Navigate to Drop-off]
    
    InTransit --> ArrivedDrop[Click End Ride]
    ArrivedDrop --> PaymentWait[Wait for Payment Confirmation]
    PaymentWait --> RateRider[Rate Rider]
    RateRider --> Standby
```

# Benefits of Detailed User Flow

* **Clear UI/UX Architecture**: Helps React developers map page transitions and conditional routing states.
* **API Route Definition**: Outlines exactly which HTTP verbs and Socket events need to be triggered at each action point.
* **Exception Handling**: Maps out logic for timeouts, payment failures, and driver rejection paths.
* **Team Alignment**: Provides a visual timeline for parallel work across frontend, backend, and testing.
